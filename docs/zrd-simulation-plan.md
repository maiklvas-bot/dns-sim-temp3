# ТЗ: Бизнес‑симуляция «Институт ЗРД: Покорение новых территорий» — v1 (тонкий срез)

> Статус: **планирование**. Реализацию не начинаем до утверждения.
> v1 = тонкий вертикальный срез, **режим ассессмента**, **solo против AI**. Мультиплеер и режим развития — следующие шаги (заложить в архитектуру, не строить в v1).
> Источники требований: профиль должности ЗРД (12 компетенций), концепт «Покорение Марса → DNS», реальная структура Дивизиона Урал.

## 1. Назначение
Отдельный интерфейс внутри проекта‑симуляции для **более высокого грейда (ЗРД — заместитель регионального директора)**. Стратегическая экономическая симуляция: ЗРД «терраформирует» регион (поднимает Продажи / NPS / Охват), управляя сетью магазинов через карты проектов, ресурсы и события. Параллельно — диагностика 12 компетенций ЗРД, **встроенная в игру**.

## 2. Зафиксированные решения
| Параметр | Решение |
|---|---|
| Режим v1 | Только **ассессмент** (строгий фикс‑сценарий) |
| Ход | **Пошаговый** (realtime держим в уме — движок не зависит от живого таймера) |
| Состав v1 | **Solo: 1 игрок против AI**. Формат поддержит и user‑vs‑users (2–4) — шаг 2 |
| Длительность | **~1 час**, ускорение игрового времени **×1–10** |
| Сложность | **1–5**, задаёт оценщик |
| Скоринг компетенций | **Встроен в игру** (алгоритм по телеметрии действий). Наблюдатель **оценивает результаты**, не ставит компетенции вручную |
| Компетенции | Профиль **ЗРД (12)** + собственный радар |
| Карта | **Реальная структура Урала**: Дивизион → РРС → магазины |
| Ресурсы | **5**: капитал, персонал, технологии, склады, потенциал рынка |
| Показатели | **3**: Продажи/выручка, NPS/сервис, Охват |
| Конец/победа | (а) три показателя достигли целевых порогов **ИЛИ** (б) истекло время → побеждает **наибольший итоговый результат** |
| Декларация цели | **Да** — игрок в начале формулирует стратегию + KPI |

## 3. Игровой цикл
**Декларация цели** (1 раз в начале) → повтор «кварталов»:
1. **Исследование** — игрок получает N карт проектов, оставляет нужные за плату.
2. **Планирование** — распределяет ресурсы/действия (внутренний шаг, без отдельной фазы у AI).
3. **Действия** — 1–2 действия за ход: разыграть проект / стандартное действие / занять/улучшить точку сети / инициировать награду.
4. **Событие** — 1 случайное (реакция игрока).
5. **Отчёт/производство** — пересчёт ресурсов и 3 показателей.
→ Финал по условию п.2 (Конец/победа).

## 4. Ресурсы (5) и показатели (3)
**Ресурсы** (производятся проектами/магазинами, тратятся на проекты):
- Капитал (мега‑кредиты) · Персонал (HR) · Технологии (CRM/BI/IT) · Склады (логистика) · Потенциал рынка (клиентская база).

**Глобальные показатели (цели терраформирования):**
- Продажи/выручка (аналог кислорода) · NPS/качество сервиса (аналог температуры) · Охват — число магазинов/население (аналог океанов).
Достижение пороговых уровней → победа; итоговый «рейтинг» (ТР) = сумма показателей + очки за проекты/инициативы.

## 5. Карта и уровни сложности (реальная структура)
Доска — **Дивизион Урал** и выбранные **РРС** с их **магазинами**. Уровень 1–5 (оценщик) регулирует масштаб и жёсткость:

| Сложн. | Зона | Чем сложнее |
|---|---|---|
| 1 | 1 компактный городской РРС (напр. «РРС Урал ЕКБ»), ~6 магазинов | лёгкая логистика, щедрые ресурсы, мягкие события |
| 3 | 1 РРС с разбросом (Пермь / Тюмень) | дальние точки, средние события, ограниченные ресурсы |
| 5 | 2–3 РРС, в т.ч. северные (Север 1/2: Сургут, Новый Уренгой, ХМ) | тяжёлая логистика, скудные ресурсы, жёсткие события |

Типы точек (MVP — 3): **обычный** магазин, **Гипер** (дороже/доходнее), **Точка выдачи доставок/склад** (логистический узел). Реальные названия — для антуража.

## 6. Карты проектов (MVP ~15, 4 категории)
- **Инфраструктура**: открыть/улучшить магазин, склад, кассовое/сервисное оборудование.
- **HR**: набор, обучение, мотивационные программы.
- **Маркетинг**: реклама, акции, локальное продвижение.
- **IT/цифровизация**: CRM, BI, автоматизация склада.
Параметры карты: стоимость (ресурсы), условия (мин. показатель/инфраструктура), эффект (рост показателя или производство ресурса), длительность, побочка (рост текучести/нагрузки).
**Стандартные действия** (всегда доступны): открыть магазин, нанять, акция, улучшить сервис — чтобы игрок не застревал при плохом раскладе.

## 7. События (3–5 в срезе)
Рыночное колебание, сбой логистики, **HR‑дилемма** (текучесть / непопулярная мера), **конфликт** (между точками), выход конкурента. Реакция игрока → влияет на показатели И даёт сигнал по компетенциям.

## 8. Декларация цели/стратегии
В начале игрок выбирает стратегию (напр. «лидер сервиса» / «экспансия» / «эффективность») и промежуточные KPI. Используется для оценки **Целеполагания** и **Стратегического видения** (соответствие действий заявленному).

## 9. AI‑бот
Rule‑based «жадная» эвристика: ранжирует доступные карты по «выгода/стоимость», занимает лучшие слоты, реагирует на события по таблице. Без ML. Уровень 1–5 влияет на качество/агрессивность бота.

## 10. Ускорение времени ×1–10
Множитель ускоряет **автоматические фазы** (производство, тики событий, «кварталы»), не нарушая пошаговость ходов игрока. Управляется игроком (и наблюдателем в просмотре).

## 11. Ядро: модель скоринга компетенций (ВСТРОЕНА)
Каждое действие порождает «сигналы», которые маппятся на 12 компетенций; из сигналов алгоритм считает балл **ФАКТ** (0–5), как в текущем радаре НАДО/ФАКТ.

| Компетенция | Механика‑крючок | Сигнал (телеметрия) |
|---|---|---|
| Планирование и организация | закуп карт, резерв ресурсов, порядок ходов, делегирование | #карт, размер резерва, последовательность |
| Постановка цели | декларация стратегии в начале | цель ↔ соответствие действий |
| Принятие решений | выбор под неопределённостью, риск, пересмотр | время на ход, смена решения после события |
| Аналитическое мышление | использование BI‑панели перед ходом | открывал отчёты, связь выбора с данными |
| Гибкость/адаптивность | реакция на событие, смена плана | изменение стратегии после события |
| Коммуникация | переговоры/обмен (мультиплеер) | число/исход переговоров (слабо в solo) |
| Ориентация на результат | контроль промежуточных порогов KPI | прогресс к целям, поведение при отставании |
| Мотивация и построение команды | HR‑проекты + кадровые дилеммы | выбор в HR‑дилеммах |
| Критическое мышление | «ловушечные» карты/события, откат | выбор в дилеммах‑ловушках |
| Инициативность | инициативы/награды сверх минимума | число инициатив |
| Управление конфликтами | события‑конфликты | выбор в конфликт‑событиях |
| Стратегическое видение | долгие vs короткие вложения | доля «долгих» проектов, стратег‑развилки |

> ⚠️ **Главный риск — валидность этой модели** (особенно «мягких» компетенций: Коммуникация / Команда / Конфликты в solo ловятся слабо). Митигация — отдельный этап проверки на реплеях (см. этапы).

## 12. Роль наблюдателя (оценщик)
Оценщик **не ставит компетенции вручную**. Он:
- задаёт **сложность 1–5** и состав;
- **наблюдает** live (read‑only доска + лента решений);
- по окончании **оценивает результаты**: смотрит готовый отчёт (радар ЗРД, лог решений, итоговые показатели/ТР), выносит вердикт/комментарий.
Переиспользуем экран «Результаты» кабинета оценщика.

## 13. Интерфейсы / экраны
- **Кабинет оценщика**: тип «ЗРД» → настройка сессии (сложность 1–5, РРС/магазины или авто, состав, «наблюдать») → код доступа → live‑наблюдение → результаты/отчёт.
- **Игрок**: вход по коду → доска (РРС/магазины) + рука карт + панель **5 ресурсов** + BI‑панель (3 показателя + отчёты) + действия хода + события + декларация цели + индикатор времени/ускорение → финал.
- **Наблюдатель (live)**: read‑only доска + лента решений + текущие показатели.

## 14. Модель данных (эскиз)
- `ZrdSession`(id, difficulty 1–5, regionScope[РРС/stores], participants, mode=assessment, status)
- `Player`(resources{5}, declaredGoal, boardState)
- `ProjectCard`(cost, conditions, effects, category, duration, sideEffects)
- `EventCard`(trigger, options[], effects)
- `TurnLog`(playerId, action, ts, signals[]) — основа и для скоринга, и для наблюдения
- `CompetencyResult`(12 × ФАКТ 0–5)
- `Outcome`(metrics{3}, TR, win, rank)
Переиспользовать существующие сессии/коды/отчёты, где возможно.

## 15. Этапы сборки среза
1. **Экономика на бумаге/в таблице** (15 карт, 5 ресурсов, 3 показателя) → плейтест баланса.
2. **Модель скоринга 12 компетенций** → проверка валидности на реплеях (эксперт согласен с ФАКТ‑профилем).
3. **Движок** (состояние, ходы, события, отчёт), solo.
4. **AI‑бот**.
5. **UI игрока + наблюдателя** + интеграция в кабинет оценщика.
6. **Пилот**, калибровка сложности 1–5 и скоринга.

## 16. Критерии готовности среза
- Партия проходима за ~1 ч (с ускорением), без тупиков.
- Баланс: жизнеспособны разные стратегии (нет одного доминирующего хода).
- Скоринг: эксперт согласен с ФАКТ‑профилем в ≥ целевого % случаев.
- Оценщик видит понятный отчёт и выносит вердикт.

## 17. Бэклог (НЕ в срезе)
Мультиплеер 2–4 (turn/realtime), тайлы‑соседство на карте, полный набор наград/достижений, редактор контента, все РРС, режим развития, умный AI, аналитика чата/переговоров.

## 18. Риски
- **Валидность скоринга** (главный) → этап 2.
- **Баланс экономики** → этап 1.
- **Мягкие компетенции** в solo почти не наблюдаемы → частично через HR/конфликт‑события; полноценно — в мультиплеере (шаг 2).
- **Скоуп** → держим срез узким.
