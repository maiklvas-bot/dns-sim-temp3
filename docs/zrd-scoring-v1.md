# ЗРД — модель скоринга компетенций v1 (Фаза 2)

Артефакт для **экспертной сверки** (точка остановки плана перед UI). Описывает, как
встроенный алгоритм превращает решения игрока в баллы ФАКТ по 12 компетенциям ЗРД,
результаты калибровки и честные ограничения. Реализация: `shared/zrd/scoring.ts`.
Проверка: `npx tsx script/zrd-sim.ts` (зелёный) и `npx tsx script/zrd-calibrate.ts`.

## Принцип (§8a — анти-хак)
Компетенция считается **по уместности выбора к контексту**, а не по тексту/форме варианта.
В момент решения движок фиксирует контекст-теги состояния (`ctxTags`: мало/много капитала,
есть персонал, отстаём от целей, движок стабилен и т.д.). Выбор события сравнивается с
тегами «когда уместен» (`option.fitsWhen`). Совпало с контекстом → сильный сигнал; выбран
заведомо слабый вариант → штраф. Числа в карточках/вариантах при оценке роли не играют —
сигнал идёт от **соответствия ситуации**.

## Крючки «сигнал → компетенция» (12)
| Компетенция | Что считается сигналом в игре |
|---|---|
| Планирование и организация | раннее построение движка (доход/долгосрочные карты в Q1–Q2), построенный prodК, активность, дисциплина добора (1–2 карты) |
| Постановка цели | декларация стратегии + консистентность сыгранных карт со стратегией |
| Принятие решений | уместные выборы событий (fit) против слабых; действие вместо паралича |
| Аналитическое мышление | использование BI-данных перед ходами + IT-карты |
| Гибкость / адаптивность | адаптивная реакция на события (колебание рынка, сбой логистики) по контексту |
| Коммуникация | совещание в конфликте точек (+ базовый уровень; в solo слабо наблюдаема) |
| Ориентация на результат | прогресс показателей к целям к концу партии |
| Мотивация и построение команды | HR-вложения + выбор «обучить» в текучести кадров (против «заменить/игнор») |
| Критическое мышление | избегание заведомо слабых вариантов событий |
| Инициативность | проактивность: число проектов, стратегические карты, декларация |
| Управление конфликтами | разрешение конфликта точек (совещание > волевое > игнор) |
| Стратегическое видение | долгосрочные карты, охват категорий, сбалансированность показателей |

## Калибровка шкалы (0–5)
`score = clamp(1 + (raw / EXPECTED_MAX) * 4, 0, 5)`. `EXPECTED_MAX` выставлены так, чтобы
**компетентная средняя игра ≈ 3.0**, слабая — заметно ниже, выдающаяся — выше. Откалибровано
эмпирически на 30 seed (L3) по персонам weak/improviser/balanced/planner (`zrd-calibrate.ts`).

Читается верно: **weak проседает по всем наблюдаемым компетенциям** (особенно команда 0.4,
аналитика 1.2, стратегия 1.5, принятие решений/результат 1.8). **planner-аналитик** выделяется
на аналитике (4.3), постановке цели (4.6), планировании. Средняя игра держится в середине (~2.9).

## Автоматические проверки (зелёные, `zrd-sim.ts`)
- weak < balanced по 5/5 «жёстких» компетенций; общий профиль weak (2.1) < balanced (2.9).
- planner ≥ balanced по аналитике/планированию/цели (3/3); разрыв planner−weak по аналитике ≥1.5.
- средний профиль balanced в полосе [2.5, 4.0]; все баллы в 0..5.

## Честные ограничения (что НЕ закрыто алгоритмом — на экспертный взгляд и Фазу 6)
1. **balanced ≈ improviser** и ряд компетенций «плоские» между всеми НЕ-слабыми персонами:
   rule-based «хорошие» боты делают почти одинаково хорошие выборы. Шкала уверенно отличает
   **слабую игру от компетентной**, но тонкое «средний vs сильный» на событийных/мягких
   компетенциях ограничено. Реальная вариативность появится на **живых игроках**.
2. **Мягкие компетенции** (Коммуникация, Команда, Конфликты) в solo ловятся слабо — лишь через
   HR/конфликт-события. Полноценно — в **мультиплеере** (шаг 2, заложено в архитектуру).
3. Это **алгоритмический первый проход**. Финальная валидность — согласие эксперта с профилями
   на **пилотных партиях живых людей** (Фаза 6), с подстройкой весов/крючков (не графики).

## Что нужно от эксперта (точка остановки)
Согласны ли вы с логикой «сигнал → компетенция» и формой профилей (слабая игра проседает,
аналитик выделяется)? Если по какой-то компетенции крючок кажется неверным — правим
`shared/zrd/scoring.ts` (веса/правила), а не UI. После «ок» — Фаза 3 (сервер/API/БД).
